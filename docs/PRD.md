PRD  
在庫管理ダッシュボード  
Inventory Dashboard — Product Requirements Document  
v0.1 · 2025年  
ステータス: Draft

## 1. 背景と課題

### 1-1. 背景
- 冊子・ステッカーなど複数の物販商品の在庫を保有  
- 提携ギャラリーが毎年委託販売を行い、1商品あたり年間 1〜10 個程度が複数商品売れる  
- 毎年 1〜3 商品（100〜200 部規模）の新作を制作  
- 売上は精算書（PDF形式）で受け取っている  

### 1-2. 課題（As-Is）
- 確定申告時に年間の在庫売上を集計する必要がある  
- 複数の精算書 PDF を目視で確認しながら手動で数量を合算しており、非常に手間がかかる  
- 精算書ごとに商品名の表記揺れが発生するため、同一商品の突き合わせも手作業  

### 1-3. 解決方針（To-Be）
精算書 PDF をそのままアプリにアップロードするだけで売上データが自動抽出・集計され、確定申告に必要な年間在庫リストを即座に確認できるようにする。  

## 2. プロダクトゴール

**分類 / 内容**

- **主目標**: 精算書 PDF から売上データを自動抽出し、年間の販売数・売上金額を一覧で確認できる  
- **副目標**: 手動入力や期間絞り込みも可能にし、将来的な在庫残数管理への拡張に耐える設計  
- **対象ユーザー**: 委託販売を行う個人クリエイター（1名）  
- **利用頻度**: 精算書受領時（年数回）+ 確定申告期（年1回）  

## 3. スコープ

### 3-1. In Scope（v1.0）
- 精算書 PDF アップロードと AI による売上データ自動抽出  
- 商品の手動追加・編集・削除  
- 精算書ごとの表記揺れを考慮した商品名の自動名寄せ（マッチング確認 UI 付き）  
- 年間（1/1〜12/31）の売上集計リスト表示  
- 任意期間（日・月・年単位）での絞り込み  
- CSV エクスポート  
- ライト / ダークモード切り替え  
- PC・SP レスポンシブ対応  

### 3-2. Out of Scope（将来拡張候補）
- 初期在庫数の登録と在庫残数（残数 = 初期在庫 − 売れた数）の管理（v2 候補）  
- 複数ユーザー・チーム利用  
- クラウド同期・ログイン機能（v1 はローカルストレージ）  
- 原画以外の一点物管理  

## 4. 機能要件

### 4-1. ダッシュボード（最優先画面）
最も重要な画面。アプリ起動時に最初に表示する。  

#### 4-1-1. 年間売上サマリー
- 選択中の期間における「総売上金額」「総販売点数」「登録商品数」をカード形式で表示  
- 期間セレクタ（年・月・日単位 or 任意範囲）をページ上部に常設  
- デフォルト期間: 現在年の 1/1〜12/31  

#### 4-1-2. 商品別売上リスト（メインテーブル）

**列定義**

- 商品名（text）: 名寄せ後の正規名称  
- 販売数量（期間合計, number）: 選択中の期間内の合計  
- 単価（number）: 精算書から取得。複数単価ある場合は最新値  
- 売上金額（期間合計, number）: 販売数量 × 単価  
- 最終販売日（date）: 期間内の直近精算書の日付  
- 登録元（badge）: PDF自動 / 手動入力 のバッジ表示  

**挙動**

- テーブルは商品名・販売数量・売上金額でソート可能  
- 商品名でのインクリメンタル検索  
- CSV エクスポートボタンをテーブルヘッダーに設置  

### 4-2. 精算書アップロード

#### 4-2-1. アップロード
- 複数 PDF の一括アップロード対応（ドラッグ & ドロップ + ファイル選択）  
- PDF フォーマット: デジタルテキスト型・スキャン画像型の両方に対応  
- スキャン PDF の場合は OCR 処理を実施（Claude Vision API を利用）  

#### 4-2-2. データ抽出
- PDF から以下の情報を抽出: 精算日・商品名・販売数量・販売単価  
- 抽出結果をユーザーが確認・修正できる「抽出確認ダイアログ」を表示してから登録  

#### 4-2-3. 例外処理
- 「原画」を含む商品名は自動的にフラグを立て、在庫集計対象外とする  
- 原画と判定した根拠（商品名）をユーザーに表示し、手動で対象に戻せる UI を提供  
- 金額・数量が読み取れない場合は該当セルを空欄にし、警告マーク付きで表示  

### 4-3. 商品名 名寄せ（表記揺れ対応）
- 同一アップロードバッチ内または既存データとの類似度を計算（編集距離 or Claude API）  
- 類似度が閾値以上の場合、「この商品は〇〇と同一ですか？」と確認 UI を表示  
- ユーザーが「同一」と判断した場合、正規名称に統一して合算  
- 名寄せの履歴（エイリアス一覧）を設定画面で確認・修正可能  

### 4-4. 手動商品登録
- 商品名・単価・販売数量・販売日を手動で入力して登録  
- 登録後はダッシュボードのリストに即時反映  
- 既存商品への追加（合算）も可能  

### 4-5. 精算書一覧
- アップロード済み精算書の一覧（ファイル名・精算日・抽出商品数）を表示  
- 個別削除（削除時は該当精算書から抽出されたデータを除外）  
- 再抽出ボタン（PDF の再パース）  

### 4-6. データ管理・エクスポート
- ダッシュボードテーブルを CSV 形式でエクスポート（表示中の期間・フィルタ条件を維持）  
- 全データのバックアップエクスポート（JSON 形式）  
- バックアップ JSON からのインポート（復元）  

## 5. 非機能要件

| 項目 | 要件 |
|------|------|
| データ永続化 | ブラウザの localStorage（v1）。将来的なクラウド移行を考慮し、データ層を抽象化 |
| PDF 解析 | Claude API（Vision）を使用。デジタル PDF は PDF.js でテキスト抽出後に渡す |
| レスポンシブ | PC（1280px〜）を基準設計。SP（375px〜）でも主要機能が使用可能 |
| テーマ | ライト / ダークモード切り替え（OS 設定を初期値として自動適用） |
| デザイン指針 | HIG / Material Design 参考。シンプル・情報密度高め。過剰な装飾を避ける |
| パフォーマンス | PDF アップロード後の抽出結果表示まで 30 秒以内（API 依存） |
| 対応ブラウザ | Chrome / Safari / Firefox 最新版 |

## 6. UI 設計方針

### 6-1. 画面構成

| 画面名 | 概要 |
|--------|------|
| ① ダッシュボード | 【最優先】年間売上サマリー + 商品別売上リスト |
| ② 精算書アップロード | PDF アップロード + 抽出確認ダイアログ |
| ③ 精算書一覧 | アップロード済みファイルの管理 |
| ④ 手動入力 | 商品の手動追加フォーム（モーダル or サイドシート） |
| ⑤ 設定 | 名寄せエイリアス管理・データバックアップ |

### 6-2. ナビゲーション
- PC: 左サイドバー固定（テキストナビ）  
- SP: ボトムタブナビゲーション（主要 3 画面）  
- 全画面共通ヘッダー: アプリ名・テーマ切り替えトグル  

### 6-3. 主要コンポーネント
- サマリーカード: 数値 + ラベル + 前期比（将来拡張）  
- データテーブル: ソート・検索・ページネーション付き  
- アップロードゾーン: ドラッグ & ドロップ対応  
- 抽出確認ダイアログ: 抽出結果の編集 + 名寄せ提案  
- 期間ピッカー: プリセット（今年・先月・先年）+ カスタム範囲  

## 7. データモデル（論理設計）

### 7-1. エンティティ

**Product（商品）**

| フィールド | 型 | 説明 |
|-----------|----|------|
| id | uuid | 一意識別子 |
| canonicalName | string | 正規商品名（名寄せ後） |
| aliases | string[] | 表記揺れ一覧 |
| latestUnitPrice | number | 最新の販売単価 |
| isExcluded | boolean | 原画など在庫集計除外フラグ |
| createdAt | date | 作成日時 |

**SaleRecord（売上レコード）**

| フィールド | 型 | 説明 |
|-----------|----|------|
| id | uuid | 一意識別子 |
| productId | uuid | Product への参照 |
| statementId | uuid | 精算書 ID（手動入力時は null） |
| quantity | number | 販売数量 |
| unitPrice | number | 販売単価 |
| saleDate | date | 精算書の精算日 |
| source | enum | pdf \| manual |

**Statement（精算書）**

| フィールド | 型 | 説明 |
|-----------|----|------|
| id | uuid | 一意識別子 |
| fileName | string | 元ファイル名 |
| statementDate | date | 精算日（PDF から抽出） |
| uploadedAt | date | アップロード日時 |
| extractionStatus | enum | pending \| completed \| error |

## 8. 将来拡張（v2 以降）

| 優先度 | 機能 |
|--------|------|
| 高 | 初期在庫数の登録 → 在庫残数（初期在庫 − 売れた数）の計算・表示 |
| 中 | クラウドDB + ログイン機能（複数端末対応） |
| 中 | グラフ・ビジュアライゼーション（月次売上推移など） |
| 低 | 複数ギャラリー・販売チャネルの管理 |
| 低 | 確定申告用フォーマットへの直接出力 |

## 9. 未解決事項・確認事項

| # | 事項 | 備考 |
|---|------|------|
| 1 | 精算書のレイアウト・フォーマットは毎年同じか | フォーマットが異なると AI 抽出精度に影響 |
| 2 | 同一精算書の再アップロード時の重複処理方針 | ファイル名・精算日ハッシュで重複検知するか |
| 3 | 商品名の正規化ルール（名寄せ閾値）はどう決めるか | 初期は手動確認 UI で対応、将来は自動化を検討 |
| 4 | v1 のデータストレージ上限 | localStorage は 5MB 程度。PDF 本体は保存しない前提 |
